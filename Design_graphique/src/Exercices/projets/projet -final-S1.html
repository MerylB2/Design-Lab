Contexte et Objectifs

1.1 Contexte pédagogique

Le module PRJ101 constitue le projet de synthèse du premier semestre (S1). Il permet aux étudiants de mobiliser et d'intégrer l'ensemble des compétences acquises dans les modules techniques du semestre.

🎯 Objectif principal
Développer une application complète de bout en bout en mobilisant tous les modules techniques
⏱️ Durée
6-8 semaines
30-50 heures de travail
👥 Modalité
Travail individuel ou en binôme selon directives

1.2 Modules concernés

Code Module Application dans PRJ101
TC101 Algorithmique & Programmation (Python/JS) Développement backend et logique applicative
TC102 Bases de données SQL Conception et administration de la BDD
TC103 Réseaux & Systèmes (Linux, TCP/IP) Déploiement sur environnement Linux
TC104 Design graphique & UI Conception et réalisation de l'interface
TC106 Anglais technique Documentation et présentation en anglais
TC107 Soft Skills (organisation, créativité) Gestion de projet et innovation

Spécifications Techniques

2.1 Technologies obligatoires

💻 Backend
Python 3.8+ OU JavaScript (Node.js 14+)

Frameworks : Flask, Django, Express.js
🗄️ Base de données
SQL obligatoire

MySQL 8.0+ / PostgreSQL 12+ / SQLite 3
🎨 Frontend
HTML5 + CSS3 + JavaScript

Frameworks optionnels : Bootstrap, Tailwind
🐧 Système
Linux

Ubuntu 20.04+ / Debian / CentOS

2.2 Fonctionnalités minimales requises

  • Système d'authentification (inscription, connexion) OU gestion multi-utilisateurs
  • Opérations CRUD complètes sur au moins 2 entités principales
  • Système de recherche/filtrage avec minimum 2 critères
  • Visualisation de données (graphique ou statistiques)
  • Au moins 1 fonctionnalité avancée (export, API, notifications, etc.)
  • Base de données avec minimum 3 tables relationnelles
  • Interface responsive (mobile et desktop)
  • Sécurité : hachage mots de passe, protection SQL injection/XSS
💡 Conseil : Commencez par un MVP (Minimum Viable Product) avec les fonctionnalités essentielles, puis ajoutez des fonctionnalités bonus si le temps le permet.

Cadrage du Projet

3.1 Types d'applications suggérés

📊 Applications de gestion ▼
  • Gestionnaire de tâches avancé : To-do list avec catégories, priorités, sous-tâches, statistiques
  • Bibliothèque personnelle : Gestion de livres, films, musique avec recherche et statistiques
  • Gestionnaire de budget : Suivi revenus/dépenses, catégories, graphiques, objectifs d'épargne
  • Carnet d'adresses intelligent : Contacts avec historique d'interactions, rappels, groupes
  • Système de réservation : Gestion de ressources (salles, matériel) avec calendrier
🛠️ Applications utilitaires ▼
  • Calculatrice avancée : Opérations complexes, historique, grapheur de fonctions
  • Convertisseur universel : Devises, unités, formats avec API temps réel
  • Gestionnaire de mots de passe : Génération sécurisée, stockage chiffré, évaluation de force
  • Outil de prise de notes : Markdown, organisation par tags, recherche, export
  • Chronomètre/Pomodoro : Multitâche, statistiques, technique Pomodoro
🎮 Applications ludiques/éducatives ▼
  • Plateforme de quiz : Création de quiz, catégories, scores, classements
  • Jeu éducatif : Mathématiques, langues avec progression et badges
  • Flashcards (répétition espacée) : Algorithme Leitner, statistiques d'apprentissage
  • Générateur créatif : Histoires, noms, poèmes avec paramètres personnalisables

3.2 Complexité attendue

⚠️ Important : Le projet doit être "simple" mais complet :
  • 5-8 fonctionnalités principales maximum
  • 3 à 6 tables en base de données
  • Application mono-page ou 3-4 pages maximum
  • 30-50 heures de développement estimé
✅ Principe clé : Mieux vaut 5 fonctionnalités excellentes qu'une application ambitieuse inachevée !

Conseils Pratiques

7.1 Stratégie de réussite

🎯 Choix du projet
Choisir un sujet qui vous passionne, avec un équilibre entre complexité et faisabilité. Privilégier 5 fonctionnalités excellentes plutôt que 10 médiocres.
📐 Conception d'abord
Passer 10-15% du temps sur la conception (MCD, wireframes, specs). Ne pas commencer à coder sans plan clair.
🔄 Développement itératif
Créer un MVP minimal qui fonctionne, puis ajouter des fonctionnalités progressivement. Tester régulièrement.
📝 Documenter au fil de l'eau
Documenter en continu, pas à la fin. Journal de bord hebdomadaire, commentaires de code au fur et à mesure.
💾 Git régulièrement
Commits fréquents (2-3 par semaine minimum) avec messages clairs. Le Git sauve des vies !
🆘 Demander de l'aide
Ne pas rester bloqué plus de 30 minutes. Consulter la doc, forums, ou contacter l'encadrant.

7.2 Pièges à éviter

⚠️ Syndrome de l'ambition excessive : Vouloir créer "le prochain Facebook". Rester réaliste !

⚠️ Technologies trop complexes : Apprendre un nouveau framework difficile pendant le projet. Rester dans sa zone de confort + 1 nouveauté max.

⚠️ Perfectionnisme paralysant : Refaire le design 10 fois. "Done is better than perfect" pour le MVP.

⚠️ Documentation reportée : Laisser la doc pour la fin = documentation bâclée garantie.

⚠️ Pas de tests : Tester uniquement à la fin = découvrir 50 bugs la veille de la deadline.

7.3 Gestion du temps

🍅 Technique Pomodoro : 25 min de travail concentré + 5 min de pause

📦 Time-boxing : Allouer un temps maximum par tâche pour éviter de se perdre dans les détails

🎯 Priorisation MoSCoW : Must have / Should have / Could have / Won't have

⏱️ Règle des 2 minutes : Tâche < 2 min = faire immédiatement, sinon planifier
💡 Conseil d'or : Prévoir 20% de temps supplémentaire pour les imprévus. Si vous estimez 40h, bloquez 48h dans votre planning !

Sécurité et RGPD

8.1 Checklist de sécurité obligatoire

  • Mots de passe HACHÉS avec bcrypt (jamais en clair)
  • Protection contre les injections SQL (requêtes préparées/ORM)
  • Protection contre XSS (échappement automatique des templates)
  • Protection CSRF (tokens dans les formulaires)
  • Validation des entrées côté serveur
  • Pas de credentials en dur dans le code (utiliser .env)
  • Sessions sécurisées avec timeout
  • Limitation des tentatives de connexion

8.2 Exemple : Hachage de mot de passe

# Python avec bcrypt from flask_bcrypt import Bcrypt bcrypt = Bcrypt(app) # Inscription hashed_password = bcrypt.generate_password_hash(password).decode('utf-8') user = User(username=username, password=hashed_password) # Connexion if user and bcrypt.check_password_hash(user.password, password): session['user_id'] = user.id return redirect('/dashboard')

8.3 Conformité RGPD

📜 Politique de confidentialité
Page accessible détaillant les données collectées, leur utilisation, la durée de conservation
✅ Consentement explicite
Cases à cocher lors de l'inscription pour accepter les conditions et le traitement des données
👤 Droits utilisateurs
Accès : voir ses données
Rectification : modifier
Effacement : supprimer
Portabilité : exporter
🔒 Minimisation
Collecter uniquement les données strictement nécessaires au fonctionnement de l'application
⚠️ Important : La sécurité et le RGPD sont des critères d'évaluation. Une application non sécurisée ou non conforme sera pénalisée, même si elle est fonctionnelle.

Préparation de la Présentation

9.1 Structure recommandée (15-20 min)

Partie Durée Contenu
Introduction 2-3 min Contexte, problématique, objectifs
Conception 2-3 min Besoins, architecture, choix techniques
Démonstration 5-7 min Demo live des fonctionnalités principales
Technique 3-5 min Code, BDD, sécurité, déploiement
Section anglais 5 min Présentation d'une fonctionnalité en anglais
REX 2-3 min Difficultés, apprentissages, évolutions

9.2 Scénario de démonstration optimal

Minute 1 : Connexion rapide avec compte de démo

Minutes 2-3 : Démonstration de la fonctionnalité principale (ajout d'élément, validation, résultat)

Minutes 4-5 : Fonctionnalités secondaires (recherche, filtres, statistiques)

Minute 6 : Points forts techniques (responsive, gestion erreurs, sécurité)

Minute 7 : Récapitulatif rapide

9.3 Checklist J-7

  • Application complètement fonctionnelle testée
  • Slides créés (15-20 slides maximum)
  • Scénario de démo préparé et répété 3 fois
  • Section en anglais rédigée et répétée
  • Plan B en cas de bug (vidéo de backup)
  • Questions potentielles anticipées
  • Matériel vérifié (laptop, adaptateurs, chargeur)
  • Données de démo cohérentes et réalistes

9.4 Gestion des questions difficiles

Question : "Pourquoi cette technologie ?"
✅ Réponse structurée : technologie choisie + raison adaptée au projet + alternative envisagée + lien avec apprentissages

Question : "Il y a un bug dans votre démo"
✅ Rester calme, reconnaître le problème, proposer solution de continuité (vidéo backup), expliquer la méthodologie de débogage

Question : "Comment gérez-vous la sécurité ?"
✅ Mentionner bcrypt, protection SQL injection, XSS, CSRF, validation serveur, références à la documentation fournie
⚠️ À éviter absolument :
  • Lire ses notes ou les slides
  • S'excuser excessivement ("désolé c'est pas fini")
  • Minimiser son travail
  • Dépasser le temps imparti
  • Paniquer en cas de bug
